# Expedition Hub PRD

Project: Expedition Hub
Version: 1.1.0
Date: 2026-03-09 (KST)
Context: 이 문서는 Expedition Hub를 구현하고 제출하기 위한 제품 기준 문서다.

## 1. 프로젝트 개요
### 1.1 문제 / 목표
- Problem: 해커톤 운영 경험은 대회 소개, 원정대 구성, 제출 준비, 평가 이해가 여러 화면과 상태에 흩어져 있어 참가자와 심사자 모두에게 비효율적이다.
- Goal: `탐색 -> 원정대 구성 -> 제출 준비 워크플로우 관리 -> 평가 이해`를 한 제품 안에서 끊김 없이 제공한다.
- Non-goals:
  - 실시간 채팅 SaaS
  - 외부 DB 기반 운영 시스템
  - 관리자 백오피스 풀스택 구현

### 1.2 제품 한 줄 정의
- Expedition Hub는 여러 해커톤에서 원정대 구성부터 제출 준비 관리, 결과 확인까지 이어지는 재사용형 해커톤 실행 포털이다.

### 1.3 타깃 사용자
- 참가자: 해커톤을 탐색하고 일정과 제출 방식을 이해해야 하는 사용자
- 원정대장: 원정대를 만들고 역할을 배분하며 제출 준비를 이끄는 사용자
- 원정대원: 모집글을 보고 합류하고 작전실에서 준비 상태와 다음 행동을 확인하는 사용자
- 심사자: 공개 URL만으로 서비스 흐름, 제출 상태, 결과 해석을 검토하는 사용자
- 운영자 관점: 다음 해커톤에도 재사용 가능한 구조를 보고 싶은 주최 측

### 1.4 Success Metrics
- 심사자가 로그인 없이 공개 페이지 기준 핵심 흐름을 3분 내 확인 가능
- 상세 페이지에서 필수 섹션 8개 모두 확인 가능
- 원정대 모집, 베이스캠프 요약, 작전실 워크플로우, 제출, 리더보드 흐름이 끊기지 않음
- 문서만 봐도 서비스 가치와 검수 구조가 이해됨

## 2. 핵심 사용자 경험
### User Story 1
- Given 사용자가 해커톤 정보를 모를 때
- When 메인 페이지에 진입하면
- Then 해커톤 보기, 원정대 찾기, 랭킹 보기로 바로 이동할 수 있어야 한다

### User Story 2
- Given 사용자가 특정 해커톤에 관심이 있을 때
- When `/hackathons/:slug`에 들어가면
- Then 개요, 안내, 평가, 일정, 상금, 팀, 제출, 리더보드를 한 화면 맥락 안에서 이해할 수 있어야 한다

### User Story 3
- Given 원정대가 제출을 준비하고 있을 때
- When `/camp`와 `/war-room/:teamId`를 사용하면
- Then 원정대 생성, 현재 단계 확인, 다음 액션 공유, 제출 준비 현황 관리를 수행할 수 있어야 한다

### User Story 4
- Given 심사자가 결과를 검토할 때
- When 공개 URL과 데모용 예시 원정대 흐름을 확인하면
- Then 별도 키 없이 서비스 가치, 제출 상태, 결과 흐름을 빠르게 검수할 수 있어야 한다

## 3. 제품 개념
### 3.1 원정대
- 해커톤에 함께 참가하는 팀이자, 제출 준비를 수행하는 실행 단위다.

### 3.2 베이스캠프
- 별도 라우트가 아니라 작전실 상단의 원정대 홈/상태 요약 영역이다.
- 표시 항목:
  - 원정대명
  - 연결된 해커톤
  - 현재 제출 단계
  - 다음 해야 할 일
  - 멤버 참여 상태

### 3.3 작전실
- 팀 작업 공간을 넘어, 체크리스트, 제출 상태, 링크 관리를 하나로 묶은 제출 준비 관리 허브다.
- 필수 포함:
  - 단계 스테퍼
  - 플랜 카드 보드
  - 링크 관리
  - 팀 메모
  - 체크리스트

## 4. 기술 스택 & 제약
### 4.1 고정 스택
- Framework: Next.js App Router
- Language: TypeScript
- Styling: Tailwind CSS
- Storage: `localStorage`
- Deploy: Vercel

### 4.2 환경 제약
- 외부 키 없는 검수 가능 상태 유지
- 더미 데이터와 로컬 상태 유지 전략 사용
- 모바일 기본 대응
- 저작권과 라이선스 준수
- 확장은 기본 구현을 해치지 않는 범위에서만 추가

## 5. 기능 명세
### 5.1 필수 페이지
- `/`
- `/hackathons`
- `/hackathons/:slug`
- `/camp`
- `/rankings`
- `/war-room/:teamId`

### 5.2 필수 기능
- 공통 상단바 이동
- 공통 상태 UI 3종
- 멀티 해커톤 목록
- 상세 필수 섹션 8개
- 원정대 모집 기본 CRUD
- 베이스캠프 상태 요약
- 작전실 워크플로우 보드
- 제출 준비/상태
- 리더보드
- 글로벌 랭킹 기본형

### 5.3 선택 확장
- 상태/태그/기간 필터
- 원정대 초대/수락/거절
- 유의사항 팝업
- 디자인 고도화

### 5.4 입력 -> 동작 -> 결과
- 원정대 생성:
  - 입력: `팀 만들기`
  - 동작: 원정대 생성 및 해커톤 연결
  - 결과: 원정대 단위 준비 흐름 시작
- 베이스캠프 상태 확인:
  - 입력: `작전실 진입`
  - 동작: 현재 단계와 다음 액션 표시
  - 결과: 팀이 무엇을 먼저 해야 하는지 이해
- 작전실 워크플로우 관리:
  - 입력: 플랜 카드 이동, 메모 입력, 링크 등록
  - 동작: 단계 상태 저장
  - 결과: 제출 준비 과정이 시각적으로 관리됨
- 제출 준비 확인:
  - 입력: 제출 섹션 또는 링크 관리 확인
  - 동작: 기획서/웹/PDF 준비 상태 반영
  - 결과: 원정대는 준비 수준을 쉽게 파악하고, 심사자는 공개 상태 요약으로 전체 흐름을 검수

## 10. Acceptance
- 제품 정의가 `Expedition Hub` 기준으로 모든 문서에서 일관됨
- 베이스캠프는 별도 새 라우트가 아니라 작전실 상단 요약 영역으로 유지됨
- 작전실 설명이 `협업 공간`이 아니라 `제출 준비 관리 허브`로 일관됨
- 심사자 검수 설명이 공개 데이터와 팀 전용 데이터 분리 원칙과 충돌하지 않음
- 필수 페이지와 필수 기능이 문서 전반에서 빠짐없이 등장함
- 심사자가 문서만 봐도 제품 의도와 검수 구조를 이해할 수 있음
